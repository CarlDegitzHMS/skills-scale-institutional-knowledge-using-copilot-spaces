# OctoAcme — Role Kickoff Checklist

## Purpose
Ensure all necessary roles are identified, engaged, and aligned at the start of a project. Use this checklist during the initiation phase and at the beginning of each major planning cycle.

---

## 1. Role Identification

- [ ] **Project Manager** assigned and available for the project duration
- [ ] **Product Manager / Product Lead** assigned with authority to prioritize
- [ ] **Business Analyst** identified or confirmed not required (document rationale)
- [ ] **UX Designer** identified or confirmed not required (document rationale)
- [ ] **Security Lead** identified and briefed on project scope
- [ ] **DevOps Engineer** identified and available for CI/CD and environment setup
- [ ] **Lead Developer / Tech Lead** assigned
- [ ] **QA / Testing representative** identified
- [ ] Key **Stakeholders** listed with names and communication preferences

---

## 2. Role Onboarding & Alignment

- [ ] All roles have access to the project board, repository, and docs folder
- [ ] Each role has reviewed the project one-pager / charter
- [ ] RACI reference (`octoacme-role-handoffs-and-raci.md`) has been shared and reviewed
- [ ] Roles & Personas doc (`octoacme-roles-and-personas.md`) shared with new team members
- [ ] Each role understands their primary deliverables and handoff points

---

## 3. Communication Cadence Agreed

- [ ] Daily standup time and channel confirmed (default: 15 min, async-friendly option documented)
- [ ] Weekly delivery sync scheduled (PM, PdM, Tech Lead, DevOps, Security Lead)
- [ ] Sprint/milestone demo schedule set
- [ ] Stakeholder update frequency agreed (default: weekly or milestone-based)
- [ ] Escalation path documented (Team → PM → Product Lead → Sponsor)
- [ ] Security incident escalation path confirmed (includes Security Lead and on-call)

---

## 4. Artifact Ownership

- [ ] **Project One-pager / Charter**: owned by Project Manager, reviewed by Product Manager
- [ ] **Risk Register**: owned by Project Manager, contributions from Security Lead and all roles
- [ ] **Backlog & Acceptance Criteria**: owned by Product Manager, authored with Business Analyst input
- [ ] **User Research / Design Specs**: owned by UX Designer
- [ ] **Technical Design Docs**: owned by Lead Developer, reviewed by Security Lead and DevOps
- [ ] **CI/CD Pipeline & Infrastructure Docs**: owned by DevOps Engineer
- [ ] **Release Notes**: owned by Project Manager, input from Product Manager and DevOps
- [ ] **Retrospective Notes**: owned by Project Manager, contributions from all roles

---

## 5. Early Risk & Dependency Check

- [ ] Initial risk list created in Risk Register
- [ ] Security Lead has been engaged for early threat identification
- [ ] Key external dependencies (APIs, third-party services, platform teams) listed
- [ ] Compliance or regulatory requirements identified (Business Analyst + Security Lead)
- [ ] Infrastructure and environment needs confirmed with DevOps Engineer

---

## 6. Definition of Done & Quality Standards

- [ ] Definition of Done agreed and documented
- [ ] Security acceptance criteria defined (Security Lead input)
- [ ] Accessibility standards agreed (UX Designer input)
- [ ] Test coverage expectations documented (QA input)
- [ ] CI checks (lint, test, security scan) enabled in the repository

---

## Notes
- Revisit this checklist at the start of any major new phase or when significant team membership changes occur.
- For small projects or short-lived initiatives, roles may be combined — document who covers which responsibilities.
